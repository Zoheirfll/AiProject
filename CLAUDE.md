## Project

GRH-Auto — HR automation platform for an Algerian HR department. Everything runs local/self-hosted to comply with Loi 18/07 (personal data must not leave national territory): no cloud LLM (Ollama only), no n8n Cloud, no external DB sync. SMTP (Gmail) is the only permitted outbound network call.

**Temporary compliance exception (dev machine, undo before any real deployment)**: `LLM_PROVIDER=groq` in `.env` currently routes all LLM calls to Groq's cloud API instead of local Ollama, because Ollama CPU inference is too slow on the dev machine while no dedicated server exists yet — this **does** send HR data outside national territory and violates the Loi 18/07 posture above. Switch back via `LLM_PROVIDER=ollama` in `.env` + `docker compose up -d --force-recreate backend` (`restart` does **not** reload `.env` changes) once a local server is available. See `agents/ollama_client.py` below for how the switch works.

Spec source of truth: `cahier_des_charges_grh_auto_v4_agile.docx` (root) — Agile/Scrum reformat of `cahier_des_charges_grh_auto_v3.docx` (kept for reference: route lists, endpoint tables, directory tree, post-MVP roadmap not restated in v4). Same scope, organized as 8 Epics (E1-E9), 34 User Stories, 254 story points, 8 one-week sprints. Sprint 1 (E1 Infra) through E8 (Temps réel & Logs) are done (E3/E4/E5/E6/E7/E8 all extended beyond or reinterpreted from spec — see below); E9 Export/Polish not started.

Note: v3/v4 assume a more granular Django app split (`imports`, `mails`, `llm`, `n8n_integration`, `notifications`, `dashboard`) than what currently exists — `automatisations` was split out in Sprint 3, `dashboard` was added for E5, and `n8n_integration` now exists too (added for E7) matching the spec's original naming; mail sending stayed in `core` (`MailLog`) rather than a separate `mails` app since it predates this reconciliation. Not fully reconciled yet.

## Stack

- Backend: Django 5 + DRF + Strawberry GraphQL + Django Channels (WebSocket)
- Frontend: React 18 + Vite + Tailwind CSS v4 + React Query (TanStack) + Recharts
- DB: PostgreSQL 16
- Scheduler: APScheduler (`django-apscheduler`, jobs stored in DB)
- AI: Ollama (local LLM, runs on the host — NOT in Docker, see Conventions) + LangChain agents (Analyste, Assistant Chat, Orchestrateur — planned). Groq cloud API is a temporary alternate provider — see compliance exception note above and `agents/ollama_client.py` below.
- Visual workflows: n8n Self-Hosted (excluded: n8n Cloud)
- Orchestration: Docker Compose

## Architecture

**Monorepo layout**: `backend/` (Django) + `frontend/` (React) + `n8n/workflows/` (exported n8n workflows) + `docker-compose.yml` at root. No shared packages between backend/frontend — communication is via REST/GraphQL/WebSocket only.

**Django apps** (`backend/`), each independent with its own `models.py`/`views.py`/`urls.py`/`admin.py`:
- `accounts` — authentication and authorization (added post-MVP, not in the original v3/v4 epics):
  - `User` (`AUTH_USER_MODEL = "accounts.User"`, `AbstractUser` + `role` field: `DRH`/`CHARGE_RH`, `is_drh` property). Introduced early enough that no other app has an FK to the user model yet — if that changes, migrations must be re-checked for the same admin/accounts ordering issue described below.
  - Session-cookie auth (not JWT/localStorage tokens) via DRF's `SessionAuthentication` — same trust model as Django admin: httponly `sessionid` cookie + CSRF-protected writes. `REST_FRAMEWORK.DEFAULT_PERMISSION_CLASSES = [IsAuthenticated]` globally in `config/settings/base.py`; every `HealthView` across all apps explicitly overrides to `AllowAny` (health checks must work unauthenticated).
  - `GET /api/auth/csrf/` (call once before login — sets the `csrftoken` cookie via `get_token(request)`), `POST /api/auth/login/` (username+password, sets `sessionid`), `POST /api/auth/logout/`, `GET /api/auth/me/` (current user or 401).
  - `LoginAttempt` model + brute-force lockout: 5 failed logins for the same **(username, IP)** pair within 15 minutes → `429` on further attempts (`accounts/views.py`'s `MAX_TENTATIVES`/`FENETRE_VERROUILLAGE`, `_client_ip()` reads `X-Forwarded-For` then falls back to `REMOTE_ADDR`); successful login clears that pair's attempt history. Keyed by IP as well as username (not just username, fixed after a security audit) — otherwise an attacker could lock out the DRH account for everyone by deliberately failing 5 logins against that known username from any source, regardless of the real DRH's own IP.
  - `accounts.permissions.IsDRH` — restricts a view/method to `role="DRH"`. Applied to: `core.ImportMappingView` (whole view), `core.ImportDeleteView` (whole view), `core.SmtpTestView` (whole view), `core.ConfigView` (whole view — fixed after a security audit found it open to any authenticated user despite returning the SMTP username), `automatisations.AutomatisationConfigView` (whole view), and via `get_permissions()` overrides to only the `DELETE` method on `automatisations.RegleDetailView` and `automatisations.TacheSurveillanceDetailView` (GET/PUT stay `IsAuthenticated`, open to both roles — a Chargé RH can edit a rule's schedule/recipients/prompt from `/automatisations`'s "Modifier" button, not just create/run/test/delete-attempt it). Chargé RH can create/run/test/edit rules and tasks but can't delete, configure the watched-folder/column-mapping, edit the global Ollama prompt/report time, view `/api/config/`, or test SMTP.
  - **Ownership-based visibility** (distinct from `IsDRH`): `RegleAutomatisation` and `TacheSurveillance` both have a `cree_par` FK (`SET_NULL`, set automatically to `request.user` on create via `perform_create()`). `automatisations.views._regle_visible_qs(user)` / `_tache_visible_qs(user)` return everything for a DRH, but for a Chargé RH only rows they created **or** ownerless legacy rows (`cree_par__isnull=True`) — e.g. a Chargé RH "recrutement"'s daily contract-expiry rule is invisible to a Chargé RH "paie", but the DRH sees both. Every automatisations/surveillance endpoint that takes a `pk` (detail, run, test, apercu, historique) does its `get_object_or_404` against this filtered queryset, not the raw model manager, so a hidden object also can't be reached by guessing its ID. `RegleAutomatisationSerializer`/`TacheSurveillanceSerializer` expose `cree_par`/`cree_par_username` (read-only) — the frontend shows "Créée par: X" on each card, but only to DRH users (`isDrh &&` in `AutomatisationsPage.jsx`/`SurveillancePage.jsx`).
  - `python manage.py create_hr_user <username> <email> --password <pw> --role DRH|CHARGE_RH` seeds/updates an account (idempotent — re-running updates password/role) — for initial/shell-access setup.
  - **Account management from the UI** (`GET/POST /api/users/`, `GET/PATCH/DELETE /api/users/<id>/`, both `IsDRH`-only): lets a DRH create/list/deactivate/delete accounts without shell access — the `/utilisateurs` page. `UserCreateSerializer` runs Django's `validate_password` (same validators as `AUTH_PASSWORD_VALIDATORS`) and hashes via `set_password()`. `UserDetailView` blocks a DRH from deactivating (`PATCH {is_active: false}`) or deleting their own account (400, checked by comparing `pk` to `request.user.id`) — prevents locking everyone out if they're the only DRH. `http_method_names = ["get", "patch", "delete"]` (no PUT — only partial updates are exposed).
  - GraphQL (`/graphql/`) stays `csrf_exempt`/unauthenticated — it only exposes a `health` field so far; revisit if real data gets exposed there.
  - **Migration footgun**: `AUTH_USER_MODEL` was swapped on a DB that already had `admin`'s migrations applied (admin's `LogEntry.user` FK creates a dependency on the user model). Fixed the migration bookkeeping by deleting `django_migrations` rows for `app='admin'`, applying `accounts` first, then `migrate admin --fake` — but faking a migration only updates Django's bookkeeping, it does **not** run the actual `ALTER TABLE`. This left `django_admin_log.user_id`'s real FK constraint pointing at the old `auth_user` table instead of `accounts_user`, causing `IntegrityError: ... violates foreign key constraint ... auth_user_id"` the first time any admin action tried to log against a real (non-superuser-created-by-createsuperuser) `accounts.User` row. Fixed with a manual `ALTER TABLE django_admin_log DROP CONSTRAINT ...; ADD CONSTRAINT ... FOREIGN KEY (user_id) REFERENCES accounts_user(id) ...`. The old `auth_user_groups`/`auth_user_user_permissions` tables (from the pre-swap default `auth.User`) are still present but empty and unused — `accounts_user_groups`/`accounts_user_user_permissions` are the real ones now. Not an issue going forward — only mattered for this one historical swap; if `AUTH_USER_MODEL` is ever swapped again on a non-empty DB, check **every** FK to the old user table, not just `django_migrations` state.
  - **Django admin** (`/admin/`) uses the same `accounts.User`/session auth as the rest of the app; a user needs `is_staff=True` (and `is_superuser=True` for full access) to log in there — `role="DRH"` alone does not grant admin access, they're independent Django concepts. Toggle via shell (`user.is_staff = user.is_superuser = True; user.save()`) since neither the `/utilisateurs` UI nor `create_hr_user` expose these flags (by design — `/admin/` is a break-glass/debugging tool, not a day-to-day surface).
  - **Security audit findings (fixed)**: `config/settings/prod.py` used to leave `SECRET_KEY` on `base.py`'s insecure dev default (`"unsafe-dev-key"`) if `DJANGO_SECRET_KEY` was unset — now `prod.py` requires it explicitly (`env("DJANGO_SECRET_KEY")`, no fallback), matching how `ALLOWED_HOSTS` was already handled. Every view was re-audited: only `HealthView`s (all apps), `CsrfView`, and `LoginView` are `AllowAny`; everything else inherits the global `IsAuthenticated` default or is explicitly `IsDRH`. No hardcoded credentials found anywhere in the codebase — `.env` (gitignored, never committed) is the only place secrets live, and `docker-compose.yml`/settings only ever reference them via `env()`/`${VAR}`. Note: rotating `DJANGO_SECRET_KEY` invalidates all existing sessions (Django signs session cookies with it) — expected, not a bug, if you ever need to rotate it.
- `core` — shared/cross-cutting concerns: Excel import, mail generation/log, global config
- `employees` — HR domain data (the business entities)
- `agents` — Ollama AI agents (E6): analyste, chat assistant, orchestrator. `agents/ollama_client.py` is the shared LLM client used across the whole project (`core` manual/bulk mail preview, `automatisations` rule-driven alerts and document-surveillance analysis, and this app's own three agents). `langchain`/`langchain-community` are listed in `requirements.txt` but deliberately unused — every agent here calls `ollama_client` directly with hand-built prompts and, where needed, a small deterministic router instead of an LLM-driven tool-executor, because a local small model is not reliable enough at structured function-calling/planning for this to be worth LangChain's abstraction cost. This mirrors the project's existing pattern (`automatisations.evaluer_regles`, `analyser_document`) rather than introducing a second paradigm.
  - `AgentConfig` is a singleton (`AgentConfig.get_solo()`, `GET/PUT /api/agents/config/`, DRH-only) holding a per-agent Ollama model override (`modele_analyste`/`modele_chat`/`modele_orchestrateur`, empty = fall back to `settings.OLLAMA_MODEL` via `config.modele(agent_name)`), `timeout_secondes`/`max_iterations` (US-E6-05), and `analyste_destinataires` (same `"email"`/`"departement:X"`/`"tous"` alias scheme as `automatisations._resoudre_liste`). `GET /api/agents/config/modeles-disponibles/` proxies `ollama.Client().list()` so the config UI can offer a real dropdown of locally-pulled models instead of free text.
  - **Analyste** (`agents/analyste.py`, US-E6-01, corrected to be subject-agnostic): `analyser_import(excel_import, lignes=None, cree_par=None)` runs automatically after every successful upload (`core.views.ImportUploadView`, passes `cree_par=request.user`) and folder-watch import (`core.services.scan_dossier_surveille`, no request user → ownerless) — both call it in a bare `try/except: pass` so the agent is best-effort and never breaks the import itself, and both now pass `parse_employee_excel`'s 4th return value (`lignes`) through. It does **not** hardcode categories like "expiring contract" or "new hire" — `_collecter_contenu()` dumps the imported rows (system fields **and** each row's `donnees_supplementaires`) as plain text, plus any import `erreurs`, and asks Ollama (via `analyser_document`, reusing the exact `ENVOYER:`/`SUJET:`/`CORPS:` contract from surveillance) with a subject-agnostic prompt to decide for itself what's noteworthy — the sheet might be about contracts, but just as easily leave balances, training, evaluations, anything. If it decides to alert, it emails `AgentConfig.analyste_destinataires` via `core.services.send_mail_log`. `POST /api/agents/analyses/lancer/` (ad-hoc button, no specific import, `cree_par=request.user`) now feeds it a live snapshot of all active employees (serialized, `donnees_supplementaires` included) instead of analyzing nothing. Every run — even a no-op one — is recorded as an `AgentAnalyse` (`excel_import` FK nullable, `decisions` JSON list — now just `anomalies`/`donnees_analysees` counts since there are no fixed categories left to report, `resume` carries the actual finding, `alertes_envoyees`), listed at `GET /api/agents/analyses/` → the `/agents/analyste` page. **Ownership-based visibility** (same convention as `automatisations`/`core.ExcelImport`, fixed after initially shipping with none): `AgentAnalyse.cree_par` (added in `agents.0002_agentanalyse_cree_par`) + `agents.views._analyse_visible_qs(user)` — DRH sees every analysis, Chargé RH sees only their own or ownerless (pre-migration/background) rows; `AgentAnalyseListView.get_queryset()` routes through it instead of a raw `.objects.all()`. `AgentAnalyseSerializer` exposes `cree_par_username`, shown as "Lancée par X" on `/agents/analyste`, DRH only. Tests: `core.tests.ImportUploadApiTests` mocks `agents.analyste.analyser_document` (class-level `@patch`) so the upload endpoint tests stay fast/deterministic regardless of whether Ollama is actually reachable — without it every upload test makes a real LLM call.
  - **Chat assistant** (US-E6-02/03): `agents/outils.py`'s `route_and_call(question)` matches French keywords against a fixed `TOOLS` dict (effectif total/par département, contrats expirant, répartition par type de contrat, règles actives) — deterministic, not LLM-routed, for the same reliability reason as above — and always returns at least one tool's output as grounding context, plus the list of tool names used (US-E6-02 "transparence", shown as badges in the UI). `agents/ollama_client.stream_chat(historique, contexte, model)` is a generator yielding response tokens from `ollama.Client().chat(..., stream=True)`. `agents/consumers.py`'s `ChatConsumer` (WebSocket, `/ws/chat/<conversation_id>/`, `conversation_id="new"` to start one) rejects unauthenticated sockets (`self.scope["user"]`, populated by `channels.auth.AuthMiddlewareStack` wrapping the websocket router in `config/asgi.py` — reads the same Django session cookie as the REST API), then for each incoming message: persists it as a `ChatMessage`, sends back `{"type": "conversation", "id": ...}` (so the frontend can adopt the new conversation's URL) and `{"type": "outils", "outils": [...]}`, streams `{"type": "token", "content": ...}` chunks, and finishes with `{"type": "done"}` (or `{"type": "erreur", ...}` on `OllamaGenerationError`) — the assistant's full reply is persisted as a `ChatMessage` only once streaming completes. Since `AsyncWebsocketConsumer.receive()` is async but Ollama's client streams synchronously, the actual work happens in `_handle()` run via `sync_to_async(..., thread_sensitive=False)`, which calls back into the consumer's async `self.send()` via `asgiref.sync.async_to_sync` — the same bridging pattern `integrations.notifications.notify()` already uses from scheduler jobs. `ChatConversation`/`ChatMessage` persist history in Postgres (US-E6-03); `GET /api/agents/chat/conversations/` (this user's own, `ChatConversation.user` scoped) and `GET /api/agents/chat/conversations/<id>/messages/` back the `/agents/chat/historique` page and conversation resume (`/agents/chat/<id>`). `GET /api/agents/chat/suggestions/` feeds the frequent-question chips shown on an empty chat. `frontend/src/lib/ChatPanel.jsx` is the shared UI (message list + streaming bubble + tool badges + input) used by both the full `/agents/chat` page and `frontend/src/lib/FloatingChatWidget.jsx` (rendered once in `AppShell`, visible on every authenticated page per US-E6-02).
  - **Orchestrator** (`agents/workflows.py`, US-E6-04, deliberately scoped down — see below): `STEP_PRIMITIVES` is a fixed dict of `slug -> (label, function)` (`verifier_contrats_expirant`, `generer_rapport`, `auditer_contrats`, `verifier_fiche_employe`, `envoyer_mail_bienvenue`, `notifier_departement`); `WORKFLOW_DEFINITIONS` maps the four predefined workflow types (Fin de mois / Onboarding / Audit contrats / Rapport hebdo) to an ordered list of these slugs. `lancer_workflow(execution, reprise_depuis=None)` runs `execution.etapes` (each `{slug, nom, statut, resultat, erreur}`) one at a time, saving `WorkflowExecution` after every step and pushing a `{"type": "workflow", "execution_id", "etapes", "statut"}` WebSocket notification via `integrations.notifications.notify()` on the existing `/ws/notifications/` channel (reused rather than adding a second WS endpoint) — this is what drives the live stepper on `/agents/orchestrateur`. On the first step that raises, the execution is marked `ECHEC` and stops; `POST /api/agents/workflows/executions/<id>/reprendre/` (DRH-only) finds that failed step's index and re-calls `lancer_workflow(execution, reprise_depuis=index)`, re-running only from there (US-E6-04 "reprise sur erreur"). **"Création de workflows personnalisés en langage naturel"** is intentionally *not* free-form code/DAG generation: `planifier_workflow_personnalise(instruction, model)` asks Ollama to choose and order a subset of the exact same `STEP_PRIMITIVES` slugs for a free-text instruction, then validates every returned slug against that fixed set before building the `PERSONNALISE`-type `WorkflowExecution` — the model can pick *which* vetted steps to run and in what order, never invent a new one. Onboarding's steps expect `execution.parametres["matricule"]`, provided by the caller.
  - Ownership: `WorkflowExecution.cree_par` is recorded but not used for visibility filtering (unlike `automatisations`'s `cree_par` — see below) since workflow launching **and reading** are both DRH-only — `WorkflowExecutionListView` has `permission_classes = [IsDRH]` (added after an audit found it missing while every sibling endpoint — launch/personnalise/reprendre — already had it, meaning any authenticated Chargé RH could read every DRH's workflow history unfiltered); `ChatConversation.user` **is** used to scope chat history strictly per-user (`ConversationListView`/`ConversationMessagesView` always filter by `request.user`).
- `automatisations` — APScheduler-driven business rules, two independent mechanisms:
  - **Contract-expiry alerts**: `RegleAutomatisation` (delays, department filter, recipients, prompt override) and `AlerteEnvoyee` (dedup guard so the same contract/delay/rule doesn't alert twice). `evaluer_regles()` checks all active rules against contracts nearing `date_fin`, generates+sends mail via Ollama+SMTP. `_envoyer_alerte(regle, contract, jours_restants, marquer_alerte=True, test_email=None)`: when `test_email` is given (from `RegleTestView`/`POST /api/automatisations/<id>/test/`), the mail is sent to that address alone (real `destinataires`/cc/bcc are bypassed) and no `AlerteEnvoyee` is recorded — this is the "envoyer vers une adresse de test" path (E3 completion). `apercu_regle(regle)` (→ `GET /api/automatisations/<id>/apercu/`) previews, without sending anything, which contracts/employees would currently trigger the rule (`deja_envoye` flag per match, resolved recipients, and the rendered prompt for the first match) — used for the pre-activation preview and the prompt-preview UI.
  - `AutomatisationConfig` is a singleton (`AutomatisationConfig.get_solo()`, pk=1, `GET/PUT /api/automatisations/config/`) holding `prompt_global` (default Ollama prompt, used as fallback wherever a rule/report has no more specific prompt — see `_rendre_prompt()`) and `heure_rapport_quotidien` (configurable daily-report time, replacing the old hardcoded 09:05 cron). `generer_rapport_quotidien()` now builds its digest content **via Ollama** (`agents.ollama_client.analyser_document()`, `forcer_envoi=True`) using `prompt_global` (or `DEFAULT_RAPPORT_PROMPT` if unset) — it used to be a hand-built f-string with no LLM involved. `verifier_rapport_quotidien()` is the scheduler tick (every 5 min, replacing the fixed 09:05 `CronTrigger`) that fires the report once per day once local time passes `heure_rapport_quotidien`, tracked via `AutomatisationConfig.dernier_rapport_envoye`.
  - **Generic document surveillance** (n8n-style): `TacheSurveillance` (watches one uploaded document — Excel/CSV/text — on an hourly or daily-at-a-time schedule, holds a free-text `prompt_analyse` instruction and a `mode_envoi` of TOUJOURS/ANOMALIE) and `ExecutionSurveillance` (audit log of every run, sent or not). `evaluer_taches_surveillance()` is the scheduler tick (every 5 min) that runs whatever task is due; `_executer_tache()` reads the file, calls `agents.ollama_client.analyser_document()` (asks Ollama to decide ENVOYER OUI/NON + draft SUJET/CORPS), and emails the resolved recipients only if warranted (or always, if TOUJOURS).
  - Both mechanisms share `_resoudre_liste()` for recipients: plain emails, `"departement:X"` (all active employees in that department), or `"tous"` (all active employees) — the last alias is surveillance-only in practice but works for `RegleAutomatisation` too.
  - Per-rule trigger history: `GET /api/automatisations/<id>/historique/` lists that rule's `AlerteEnvoyee` rows (employee name, contract date_fin, delay, timestamp) — distinct from the global `/api/mails/historique/` log.
- `integrations` — everything that talks outward: APScheduler config (`integrations/scheduler.py` — registers `evaluer_regles` at 09:00, `verifier_rapport_quotidien` every 5 minutes (configurable time, see above), `evaluer_taches_surveillance` every 5 minutes, `scan_dossier_surveille` every 45s), WebSocket consumer for real-time notifications (`integrations/consumers.py` + `routing.py`), `integrations/notifications.py` (`notify(payload)` — sync helper wrapping `channel_layer.group_send` so non-async code, e.g. scheduler jobs, can push to the `notifications` WS group), and (planned) n8n webhook endpoints
- `dashboard` — read-only aggregation endpoints for the `/` dashboard page (E5), no models of its own:
  - `GET /api/dashboard/kpis/` — mails sent today/this week/this month (`MailLog.sent_at`), active-rules count, contracts expiring within 30 days, Ollama reachability (`agents.ollama_client._client().list()`, wrapped in try/except), n8n reachability (`requests.get(settings.N8N_URL)`, also try/except — `N8N_URL` defaults to `http://n8n:5678`, the Docker Compose service name, overridable via `.env`'s `N8N_INTERNAL_URL`).
  - `GET /api/dashboard/mails-evolution/?jours=30` — daily sent-mail counts for the trend line chart.
  - `GET /api/dashboard/automatisations-types/` — sent-mail counts bucketed by originating mechanism (Alertes contrats / Surveillance documents / Rapport quotidien / Manuel), inferred from `MailLog.regle` and `sujet_demande`, for the pie chart.
  - `GET /api/dashboard/contrats-par-mois/` — contract counts for the next 12 months by `date_fin` month, for the bar chart.
  - `GET /api/dashboard/activite/?limite=20` — unified recent-activity feed merging sent `MailLog`s, `ExcelImport`s, `TacheSurveillance` runs, and `AlerteEnvoyee` triggers, sorted by timestamp desc, each with a `lien` (frontend route) for the "go to detail" link. `ExcelImport`/`TacheSurveillance`/`AlerteEnvoyee` entries are filtered through `core._import_visible_qs`/`automatisations._tache_visible_qs`/`_regle_visible_qs` (fixed after a security audit found this feed bypassing ownership filtering — a Chargé RH could see another Chargé RH's otherwise-hidden rule/task surface here). `MailLog` entries stay unfiltered (mail history has no ownership concept anywhere in the app, by design).
  - Real-time: `core.views._envoyer_mail_log()`, `automatisations.services._envoyer_alerte()` (only when `marquer_alerte=True`, i.e. not on test sends), and `_finaliser_execution()` (surveillance, only when `envoye=True`) all call `integrations.notifications.notify()` after a real send — the frontend dashboard listens on the same `/ws/notifications/` socket (`frontend/src/lib/useNotifications.js`) and invalidates its React Query caches on any message, so KPIs/charts/activity feed update live without polling. `core.services.scan_dossier_surveille()` already did this for imports (E2).

- `n8n_integration` (E7) — dedicated endpoints for n8n to call into GRH-Auto, secured by **scoped, DB-backed bearer tokens** instead of the session/CSRF the rest of the app uses (n8n is a machine caller, not a browser):
  - `n8n_integration.models.N8nApiToken` (`nom`, auto-generated `token` via `secrets.token_urlsafe(32)`, `scopes` JSONField — subset of the `Scope` TextChoices `employes:read`/`contrats:read`/`mails:send`/`logs:write` — `actif`, `cree_par`, `derniere_utilisation`) replaces a single global static token: a token minted for one workflow (e.g. "rapport quotidien") can be limited to just the scopes it needs and revoked (`actif=False`) independently of every other workflow's token, so a leaked or compromised token doesn't grant full access to every `/api/n8n/*` endpoint. **Mail generation (Phase 3 + Sprint 3 + E4 completion)**: `agents/ollama_client.py` wraps the `ollama` Python client (`ollama.Client(host=settings.OLLAMA_BASE_URL)`).

**Logs techniques (US-E8-02)**: `core.models.TechnicalLog` (level INFO/WARNING/ERROR, logger_name, message, created_at) is populated by `core.logging_handlers.DatabaseLogHandler`, a custom `logging.Handler` wired into `LOGGING` in `config/settings/base.py` alongside a `TimedRotatingFileHandler` (daily rotation, 14-day retention, `backend/logs/grh-auto.log` — gitignored) and a console handler. Single-recipient modes (Employé/Adresse libre) have a Texte/HTML format selector; when HTML, the draft view shows both the editable HTML source (`Textarea`) and a live rendered preview (`dangerouslySetInnerHTML`, sanitized via `DOMPurify.sanitize()` before rendering — added after a security audit flagged unsanitized LLM-generated HTML as a self-XSS-adjacent risk if a crafted prompt ever produced a `<script>` tag) below it. **Design system**: `frontend/src/theme.js` (color tokens — primary=blue/accent=emerald scales matching the hexagon/workflow logo, status-tone mapping, chart palette, shadow tokens — single source of truth, mirrored into Tailwind's `@theme` block in `index.css`) + `frontend/src/lib/ui.jsx` (reusable primitives: `Card`, `Button` (gradient fill, press feedback), `Badge`, `Modal` (entrance animation, respects `prefers-reduced-motion`), `Field`, `Input`/`Textarea`/`Select` (properly merge a caller's `className` with the base style — this used to silently discard it since `{...props}` was spread after a fixed `className`, fixed after it broke a password-toggle button's padding), `TagInput` (chip-based array editor — Entrée/virgule to add, click-to-remove — replaces raw comma-separated text fields for list-shaped values like `delais_jours`/`destinataires`), `DropdownMenu` (click-outside-to-close `⋯` menu for grouping secondary per-row actions off a card so a primary action can stay visually dominant), `HelpBanner`/`useHelpBanner(id)` (dismissible per-page info banner — dismissal persists in `localStorage` keyed `grh-auto:help-dismissed:<id>`; pair with `PageHeader`'s `onHelp`/`helpVisible` props, which render a "?" `IconButton` that calls the hook's `reopen()` once the banner has been dismissed — every authenticated page under `frontend/src/pages/` has one except `/login`), `Toast`, `EmptyState`, `Spinner`, inline icon set — no external icon library). Sidebar layout (not a top navbar), gradient active-nav-item pill. Dark mode via Tailwind v4's `@custom-variant dark (&:where(.dark, .dark *))` in `index.css` + `frontend/src/lib/useTheme.js` (toggles a `.dark` class on `<html>`, persists to `localStorage`, defaults to OS preference) — toggle button lives in the sidebar header. Font stack is native system fonts (`-apple-system, Segoe UI, …`), deliberately not a Google Fonts `@import` — that would be an outbound network call, which conflicts with this project's local-first/Loi 18/07 posture (see Project section above). Every page under `frontend/src/pages/` has had at least one design-polish pass applying this system (gradient buttons/cards, EmptyState, badges, responsive grids, dark-mode parity) — if a page still looks inconsistent with the rest, it's a regression, not the original state. `/automatisations` additionally got an ergonomics pass: `RuleCard`'s secondary actions (Aperçu/Historique/Tester/Modifier/Supprimer) moved into a `DropdownMenu` so "Exécuter" reads as the one primary action; a per-card `ConcernedBadge` fetches `fetchRegleApercu` (5min `staleTime`) to show "N concerné(s) maintenant" without opening the Aperçu modal; a search+status filter bar sits above the grid; the page's description now pulls the real `heure_rapport_quotidien` from `AutomatisationConfig` instead of a hardcoded "9h".

**GraphQL**: single schema at `backend/config/schema.py`, mounted at `/graphql/` via `config.urls.AuthenticatedGraphQLView` (subclasses `AsyncGraphQLView`, overrides `dispatch()` to 401 when `request.user.is_authenticated` is False before any resolver runs — session cookie, same trust model as the REST API), wrapped in `csrf_exempt` (POST-only GraphQL, no state-changing GET). Added after a review flagged the view as originally fully anonymous with no per-field auth convention — blocking at the view level means any field added to `Query`/`Mutation` later is covered automatically, no opt-in required. Only a `health` test field exists — no models are exposed via GraphQL yet. `strawberry_django` is intentionally NOT in `INSTALLED_APPS`: it was removed because the installed `strawberry-graphql-django==0.44.1` pulls in a `strawberry-graphql` version incompatible with `strawberry.auto` (`ModuleNotFoundError: No module named 'strawberry.auto'`) — pin compatible versions together before re-adding it.

**Media uploads**: `MEDIA_ROOT = backend/media/`, served via `MEDIA_URL = "media/"` — not committed to git (`media/` in `.gitignore`). Includes `imports/` (employee Excel uploads) and `surveillance/` (`TacheSurveillance.fichier`, re-read on every scheduled run).

**ASGI/WebSocket**: `config/asgi.py` routes HTTP through Django's ASGI app and WebSocket through `integrations/routing.py` → `NotificationsConsumer`, which broadcasts to the `notifications` channel group via Django Channels (Redis-backed channel layer). `NotificationsConsumer.connect()` rejects unauthenticated sockets (`scope["user"].is_authenticated`, same `AuthMiddlewareStack`-populated check as `agents.ChatConsumer`) — fixed after a security audit found it originally accepted any connection with no auth check at all, meaning anyone reaching `/ws/notifications/` (externally reachable via the ngrok dev tunnel, see below) got every real-time event broadcast (mail sends/failures, contract alerts, Ollama/Groq error details, workflow steps) with no login required. **`"daphne"` must be the first entry in `INSTALLED_APPS`** (`config/settings/base.py`) — Channels' `daphne` package is only wired into `manage.py runserver` (patching it into an ASGI-capable dev server) if the `daphne` app is registered before `django.contrib.staticfiles`; if it's missing, `runserver` silently falls back to plain WSGI and every `/ws/...` request 404s as an ordinary unmatched URL (logged as `django.request: Not Found`, not a Channels rejection) instead of reaching the consumer — this exact regression shipped once (daphne was in `requirements.txt` but never added to `INSTALLED_APPS`) and broke live notifications, the dashboard's live-refresh, and the chat WebSocket app-wide with no error surfaced anywhere except the browser console.

**Docker network**: services reference each other by service name (`db`, `redis`, `n8n`), not `localhost`, inside the Compose network — `.env` values like `POSTGRES_HOST=db` depend on this. Ollama is the exception (see Conventions below).

**n8n**: self-hosted only, backed by the same PostgreSQL instance (separate schema via `DB_TYPE=postgresdb` env vars in `docker-compose.yml`), workflows persisted to `n8n/workflows/`.

## Conventions

- Env config is read via `django-environ` from the root `.env` file (`backend/config/settings/base.py` points to `BASE_DIR.parent / ".env"`), not `backend/.env`.
- **Settings are split by environment**: `backend/config/settings/{base,dev,prod}.py` (not a single `settings.py`). `dev.py` reads `DJANGO_DEBUG`/`DJANGO_ALLOWED_HOSTS` from `.env` with permissive defaults; `prod.py` forces `DEBUG=False`, requires `DJANGO_ALLOWED_HOSTS` explicitly (no default — raises if unset), and enables secure cookie flags. `manage.py`/`wsgi.py`/`asgi.py` all default `DJANGO_SETTINGS_MODULE` to `config.settings.dev`; override it (env var or `docker-compose.yml`) to run against `prod`.
- Locale is fixed to `fr-fr` / `Africa/Algiers` in Django settings.
- Tailwind v4 uses the new CSS-first config (`@import "tailwindcss"` in `index.css`, `@tailwindcss/postcss` plugin) — no `tailwind.config.js`. Custom tokens/dark-mode variant also live in `index.css`'s `@theme`/`@custom-variant` blocks (see Design system above).
- **Ollama runs on the host, not in Docker** (deliberately removed from `docker-compose.yml` — the containerized image + model pull consumed too much disk). The `backend` service reaches it via `OLLAMA_BASE_URL=http://host.docker.internal:11434` (set directly in `docker-compose.yml`'s `backend.environment`, overriding whatever is in `.env`), with `extra_hosts: host.docker.internal:host-gateway` so that resolves inside the container. Start Ollama locally (`ollama serve`, or the desktop app) before running the backend. `.env`'s `OLLAMA_MODEL` must match a model you've actually pulled locally (`ollama list` to check) — it currently is NOT `llama3` (the cahier des charges default) but whatever is set in `.env`, since `llama3` isn't pulled on the dev machine. `.env.example`'s `OLLAMA_BASE_URL` reflects the non-Docker (host-run) default and is only used if you run the backend outside Docker — the Docker Compose override always wins inside containers.
